# 桌宠连接设计

## 定位

桌宠连接不是替代微信、QQ 或 Discord，而是做一个“桌面上的关系型传话和陪伴系统”。它的价值来自三个点：

- 消息由对方的桌宠表达，天然更轻、更可爱、更不正式。
- 互动发生在桌面边缘，不需要打开完整聊天窗口。
- 恋人或朋友之间可以看到彼此桌宠的在线状态和小情绪，形成低压力陪伴感。
- 用户可以把真实关系里的对方形象做成桌宠，让“桌面上的角色”背后真的连接着那个人。

## 用户故事

我把恋人或朋友的形象做成桌宠，并添加了对方的真实桌宠连接。想给对方发一句话时，我通过自己的桌宠发送。如果对方在线，对方的桌宠会实时收到消息，播放一个到达动作，并用气泡显示内容。

典型场景：

- 情侣互相把对方做成桌宠，各自放在自己的电脑桌面上。
- 一方发来短消息时，另一方看到的是“对方的形象”出现在桌面气泡里。
- 用户可以给这个连接设置关系称呼，例如恋人、好友、搭子、自定义称呼。
- 桌宠连接带来的重点不是聊天效率，而是“对方在我的桌面上陪我一下”的感受。

## 体验原则

- 桌宠仍然是主角，聊天界面要保持轻量。
- 消息要有趣，但必须清楚可读。
- 收到消息时应该能被注意到，但不能抢走用户正在做的事。
- 用户需要能控制谁可以联系自己，以及什么时候允许打扰。
- 离线时要优雅处理：排队、重试，或提示朋友暂时不在线。

## 关键对象

- 用户：真实使用产品的人，拥有账号、昵称和隐私设置。
- 我的桌宠：用户当前展示在桌面的宠物形象和资料。
- 关系桌宠：恋人、好友或重要的人在我桌面上的角色化形象，可以显示在线状态、昵称、关系称呼和可互动能力。
- 会话：我和某个朋友桌宠之间的轻量消息通道。
- 消息：短文本为主，后续可扩展为动作、贴纸、状态和宠物来访。
- 状态：在线、离开、勿扰、离线，以及用户自定义状态。
- 关系称呼：用户本地给对方设置的关系标签，例如恋人、好友、搭子、同事或自定义。

## 好友添加流程

MVP 推荐使用邀请码或邀请链接，降低账号体系初期复杂度。

1. 用户在“桌宠连接”里生成邀请。
2. 邀请包含用户昵称、桌宠名、短期有效的邀请码。
3. 朋友输入邀请码或打开邀请链接。
4. 双方确认后成为好友或关系连接。
5. 对方出现在“关系桌宠”列表中，并显示在线状态。
6. 用户可以给对方设置本地关系称呼，并选择是否把对方形象绑定为桌宠角色。

产品细节：

- 默认必须双向确认，避免陌生人直接打扰。
- 邀请码应该有过期时间。
- 用户可以随时撤销邀请、删除好友、拉黑好友。
- 关系称呼默认只保存在本地，不应未经同意展示给对方。

## 消息体验

发送侧：

- 用户点击自己的桌宠或托盘入口，打开轻量发送面板。
- 选择朋友桌宠，输入短消息。
- 发送成功后，我的桌宠可以做一个“递出消息”的小动作。
- 如果朋友离线，展示“朋友暂时不在线，可稍后送达”。

接收侧：

- 对方桌宠在线时，播放“收到消息”动作。
- 消息以气泡显示在桌宠上方，并可保留几秒。
- 如果消息较长，可以折叠为摘要，点击后打开小消息面板。
- 如果用户处于勿扰状态，只显示托盘提示或静默入收件箱。

消息限制：

- MVP 只支持短文本，建议 140 字以内。
- 不做已读压力，最多做“已送达/未送达”。
- 默认不连续弹出多条气泡，避免刷屏。

## 在线状态与存在感

状态应该表达“朋友是否方便被打扰”，而不制造强社交压力。

- 在线：桌宠正在运行，可以实时收到消息。
- 离开：桌宠在线但用户近期无操作。
- 勿扰：接收但不弹气泡。
- 离线：桌宠未连接，消息可进入离线队列。

后续可以加入轻量情绪状态：

- 专注中
- 休息中
- 想聊天
- 今天有点累

这些状态可以由用户手动设置，也可以结合本地行为轻量推断，但必须允许关闭。

## 隐私与安全

桌宠连接一开始就需要隐私边界，不能等用户量变大后再补。

- 好友必须确认后才能发消息。
- 用户可以关闭所有入站消息。
- 支持勿扰时间段。
- 支持删除好友和拉黑，服务端必须阻止已移除或已屏蔽关系继续发消息。
- 支持举报恶意消息。
- 默认不公开用户真实身份，只显示昵称和桌宠资料。
- 桌宠资料、在线状态、消息通知都应有可见性设置。
- 如果桌宠形象基于真实人物，必须允许用户删除、替换和停止展示，不应强制同步给对方。

推荐默认值：

- 只允许好友发消息。
- 陌生人不能搜索到用户。
- 勿扰时不弹出桌面气泡。
- 离线消息默认保存有限时间。

## 消息留存策略

这个产品的气质更接近轻量传话，不建议一开始做完整聊天历史。

MVP：

- 最近消息可以在本地保留。
- 服务端只保留未送达消息和必要投递状态。
- 已送达消息可以短期保留用于同步，之后清理。

后续：

- 用户可选择是否保留历史。
- 支持导出或清空本地消息。
- 对私密消息支持“阅后即消失”模式。

## 产品入口

- 桌宠点击面板：发送消息、查看朋友、设置状态。
- 偏好设置新增“连接”页：账号、好友、隐私、勿扰。
- 托盘菜单：连接状态、快速勿扰、打开朋友列表。
- 收到消息气泡：点击打开轻量会话。

## MVP 分期

第一阶段：本地模拟

- 不接真实网络。
- 在本地模拟一个朋友桌宠消息到达。
- 验证气泡、动画、勿扰、消息面板的体验。

第二阶段：账号与好友

- 用户注册或匿名设备账号。
- 邀请码添加好友。
- 好友列表和在线状态。

第三阶段：实时消息

- 好友之间短文本实时投递。
- 离线队列。
- 托盘和气泡通知。

第四阶段：宠物互动

- 发送动作。
- 轻量贴纸。
- 朋友桌宠短暂来访。
- 情侣或好友专属动作，例如挥手、比心、敲敲桌面、递纸条。

第五阶段：生态扩展

- 分享宠物包。
- 好友间推荐角色。
- 小型群组角落。

## 成功标准

桌宠连接成功与否，不看发送了多少长消息，而看它是否带来“朋友在身边一点点”的感觉。

可观察指标：

- 用户是否愿意保持桌宠在线。
- 好友添加后是否有持续轻量互动。
- 用户是否愿意把恋人或朋友做成桌宠并长期保留在桌面。
- 消息是否造成打扰投诉。
- 勿扰和隐私设置是否易于理解。
- 用户是否会主动分享邀请链接。

## 明确不做

MVP 不做这些功能：

- 大群聊。
- 语音/视频通话。
- 复杂文件传输。
- 公开陌生人搜索。
- 强已读、输入中、在线追踪压力。
- 大而完整的聊天历史管理。

## 当前实现范围

桌宠连接第一版已经从本地模拟升级为真实链路：

- Node.js Pet Connect 服务端，提供注册、邀请码、好友关系、在线状态、实时消息和离线队列。
- Electron 客户端通过 WebSocket 连接服务端。
- 偏好设置里有连接页：服务器地址、连接昵称、状态、入站消息开关、勿扰、注册/连接。
- 用户可以生成邀请码，朋友输入邀请码后建立好友关系。
- 用户可以选择好友并发送短消息。
- 好友在线时实时收到消息，桌宠展示气泡并触发轻量动作。
- 好友离线时服务端暂存消息，下一次上线后送达。
- 勿扰时客户端静默记录来信，不弹气泡。
- Privacy 页面支持隐藏在线状态，朋友会看到离线状态但连接身份仍保留。
- Privacy 页面支持删除关系和屏蔽关系；服务端移除双方好友关系，并阻止已删除或已屏蔽对象继续发消息。
- 连接页支持创建一个真实的本地测试朋友桌宠账号，用完整服务器链路测试加好友、状态和来信反馈。
- 连接页支持复制邀请码，方便发给朋友测试。
- 产品方向已明确为关系型桌宠：情侣、好友或重要的人可以互相把对方形象做成桌宠，并通过真实连接互相传话。

当前部署：

- 服务入口：`https://app.qimozhu.abrdns.com/pet-connect`
- WebSocket：`wss://app.qimozhu.abrdns.com/pet-connect/ws`
- 健康检查：`https://app.qimozhu.abrdns.com/pet-connect/health`
- VPS 部署目录：`/opt/services/pet-connect`
- 服务存储：当前为轻量 JSON 文件，挂载在服务目录的 `data` 下。

当前限制：

- 账号是设备本地注册身份，没有手机号/邮箱登录。
- 邀请码添加、好友删除、屏蔽和隐藏在线已经可用，但还没有完整的账号恢复和解封管理。
- 消息历史只有本地最近记录，没有完整会话窗口。
- 服务端使用轻量 JSON 文件存储，适合小规模朋友试用，不适合大规模公测。

下一步：

- 两台真实电脑 smoke test，覆盖安装、邀请码、互发消息、隐藏在线、删除和屏蔽。
- 设计轻量消息面板，查看最近来信和发送记录。
- 增加邀请链接分享。
- 增加服务端管理、备份和账号恢复策略。
- 评估从 JSON 文件迁移到 SQLite。
